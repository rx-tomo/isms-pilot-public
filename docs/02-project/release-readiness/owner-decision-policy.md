# 事業オーナー判断ポリシー

この文書は、release readiness 評価と改善ループ中に発生しやすい事業判断を、あらかじめ方針化するためのものです。

Codex は、ここに記載された既定方針の範囲内であれば都度停止せずに評価・課題化・軽微な修正を進めます。既定方針を超える判断が必要な場合は、実装せず「判断待ち」として記録します。

## 既定方針

| 判断領域 | 既定方針案 | Codexの扱い |
| --- | --- | --- |
| MVP範囲 | ISMS新規導入と継続運用の主要導線を優先。高度な外部連携やAI高度化は後回し可 | 主要導線の不足はP0/P1。高度機能の不足はP2/P3に分類 |
| 規格対応 | ISO原文の完全代替ではなく、運用支援・証跡管理ツールとして扱う | 規格本文を長文引用せず、要求カテゴリ、証跡、確認観点へ要約 |
| セキュリティ | テナント分離・認証・認可・監査ログは妥協不可 | 欠陥はP0/P1。修正可能なら改善候補、方針判断が必要なら停止理由へ記録 |
| データ削除 | 物理削除/論理削除/保持期間は事業方針として明示が必要 | 実装差分は未確認/判断待ちとして記録し、勝手に削除仕様を決めない |
| 課金・プラン | 実装済み範囲を優先し、自動アップグレード等は未実装なら課題化 | 料金やプラン変更は実装しない。現状とdocsの差分を記録 |
| AI機能 | AI判断を最終判断にせず、人手確認と監査ログを前提にする | AI出力の自動確定は避ける。人手確認・ログ不足は課題化 |
| リリース判定 | 90%以上かつ必須ゲート合格でリリース候補。未達なら評価レポートで停止 | 90%未達でも自動で無限ループしない。最大2回の改善で停止 |

## 事前確認したい論点

以下は、改善フェーズに入る前に事業オーナーが確認すると判断ぶれを減らせます。

| 論点 | 推奨方針 | 未確認時の扱い |
| --- | --- | --- |
| ISMS新規導入の最小完了状態 | 組織スコープ、ユーザー、文書、資産、リスク、管理策、タスク、監査計画までを最小完了とする | この導線が欠ける場合はP1以上 |
| 継続運用の最小完了状態 | 月次レビュー、承認、是正、通知、証跡エクスポートが回ること | 未実装または未検証ならP1/P2 |
| SaaS運営者の最小完了状態 | テナント作成、招待、権限、監査ログ、契約状態確認ができること | テナント作成不能や越境リスクはP0 |
| 外部連携 | Slack/Teams/SSO/高度Webhookはリリース後改善でも可 | 未実装ならP2/P3。ただし既存機能が壊れていればP1 |
| AIリスク評価 | AIは補助。最終判断は人間が確認する | AIのみで業務判断が確定する設計ならP1 |
| データ保持・削除 | 契約終了後保持期間、論理削除、物理削除、バックアップ削除は明文化が必要 | 実装しない。判断待ちにする |

## Codexが止まる条件

- 規格解釈や認証範囲の表明が、製品の法務・営業表現に影響する場合
- 顧客データ削除、保持期間、物理削除に関わる実装変更が必要な場合
- 料金、プラン、課金条件、返金、利用上限に関わる変更が必要な場合
- セキュリティ上の重大懸念があり、修正方針が複数あり事業影響が異なる場合
- AI機能の自動判断範囲を拡大する必要がある場合

## Codexが止まらず進めてよい条件

- docsと実装の差分整理
- 採点、証跡収集、未確認事項の記録
- テスト、lint、typecheck、build、QAの実行
- 明らかなバグ修正、テスト追加、文言補正、軽微なUX改善
- 既定方針に沿ったP0/P1/P2/P3分類

## レビュー時の見方

採点後のレビューでは、次の順で確認します。

1. 必須品質ゲートに失敗していないか
2. 90%未満の主要原因は何か
3. P0/P1が事業判断なしに修正できるか
4. 事業判断待ちの項目がリリース可否に影響するか
5. 最大2回の改善ループで到達できそうか
