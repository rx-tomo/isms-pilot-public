# Capability-based Release Readiness Method

このドキュメントは、ISMS Pilot の release-readiness 作業で偶然に近い形で組み上がった進め方を、後から再利用・検証・skill化できるように整理したものです。

現時点の呼び名は、次のどちらかを候補とする。

- 日本語: 必要能力ベースのリリース準備度評価
- 英語: Capability-based Release Readiness Assessment
- 別名: Release Readiness Assessment with Capability Mapping

## 何をする方法か

製品が「機能として存在しているか」だけでなく、「実務ワークフローの中で使える状態か」「証跡を残して説明できるか」「リリース前に潰すべき不足が見えているか」を確認する方法である。

特に、ISMS、セキュリティ、監査、法務、業務運用のように、画面やAPIがあるだけでは十分と言えない領域に向いている。

## 背景

今回の ISMS Pilot では、前段階として既存コードベースを調査し、`docs/01-business/spec-dsl/` に現行仕様をドキュメント化した。

その後、`docs/02-project/release-readiness/` で以下を組み合わせて段階的に評価している。

- 実務ワークフロー整理
- 必要能力、つまり Capability の定義
- Fit & Gap 分析
- リリース準備度評価
- 品質ゲート
- P0/P1/P2/P3 の優先度分解
- PDCA 単位の改善計画
- 証跡ベースの業務ジャーニー検証

この組み合わせは、単なるチェックリストではなく、仕様、実装、実務、品質、事業判断をつなぐ評価の型として扱える可能性がある。

## 基本用語

| 用語 | 意味 |
| --- | --- |
| CAP | Capability。製品が実務を成立させるために必要な能力 |
| W | Workflow。利用者や運営者が実際に行う業務の流れ |
| Fit | 必要能力に対して、現行実装や運用証跡が合っている状態 |
| Gap | 必要能力に対して、不足、未検証、判断待ち、過剰候補がある状態 |
| P0 | リリース前に必ず解消すべき重大課題 |
| P1 | MVPとして実用化するために優先して解消すべき課題 |
| P2 | リリース候補後、または改善サイクルで対応する課題 |
| P3 | 後続改善、効率化、拡張として扱う課題 |
| Gate | リリース候補にする前に満たすべき最低条件 |
| Evidence | 実装、テスト、ログ、操作結果、レビュー記録などの説明可能な証跡 |

## 入力

この方法を使う前提として、次の情報があると再現しやすい。

| 入力 | 役割 |
| --- | --- |
| 既存コードベース | 実装済み機能、API、DB、UI、テスト、スクリプトを確認する対象 |
| `docs/01-business/spec-dsl/` | 現行仕様、業務ルール、データ構造、例外条件の棚卸し |
| 既存docs | 仕様、設計、QA、運用、過去判断の確認 |
| 規格・実務ナレッジ | ISO/NIST/IPA/METI、市場慣行、運用ノウハウなど |
| 事業判断ポリシー | 判断待ちにすべきものと、既定方針で進めるものの境界 |

## 手順

### 1. 現行仕様を固定する

まず、実装、既存docs、テスト、スクリプトを読み、現時点の製品が何を持っているかを整理する。

この段階では、良し悪しを判定しすぎず、現行コードベースの仕様を `spec-dsl` のような形で取り出す。

### 2. 実務・規格・市場の観点を足す

次に、製品が扱う領域の実務や規格を調査する。

ISMSであれば、単にチェックシートを作れるかではなく、リスク評価、管理策選定、証跡、内部監査、是正、継続改善、SaaS運営上の責任分界までを見る。

### 3. 業務ワークフローを定義する

利用者、管理者、運営者が実際にどう動くかを W-01 のような単位で定義する。

この工程により、「機能一覧」ではなく「実務が最後まで流れるか」を評価できる。

### 4. 必要能力を CAP として定義する

各ワークフローを成立させるために必要な能力を CAP-01 のように分解する。

CAP は、画面名やテーブル名ではなく、実務上できる必要があることとして書く。

例:

- リスクを登録できる
- 管理策を選定し、理由を残せる
- 内部監査の結果と是正を追跡できる
- テナントごとにデータが分離される

### 5. Fit & Gap を棚卸しする

CAPごとに、現行実装、docs、テスト、DB、API、画面、証跡を確認する。

判定では、単にファイルや画面があることだけを達成扱いにしない。実務ワークフロー上、利用者が迷わず使え、結果を説明できるかを見る。

分類例:

- 実装済み
- 実装あり未検証
- docsのみ
- 不足
- 判断待ち
- 過剰候補

### 6. 優先度を分ける

Gap を P0/P1/P2/P3 に分ける。

P0/P1 は「いつかやる改善」ではなく、リリース候補にする前、またはMVPとして使える状態にする前に向き合うものとして扱う。

### 7. 品質ゲートで止める

総合点が高くても、以下のようなゲートに失敗している場合はリリース候補にしない。

- 高/重大セキュリティ問題がない
- テナント越境アクセスがない
- 認証・認可の重大不備がない
- 主要業務ジャーニーが成立している
- P0/P1 不具合が未処理で残っていない

### 8. PDCA単位に分解する

不足を一気に実装リストへ落とすのではなく、Plan、Do、Check、Act の単位で進める。

特に compliance や業務系のプロダクトでは、調査、設計、実装、証跡確認、過剰機能整理を分けることで、途中から再開しやすくなる。

### 9. 停止判断を記録する

各段階の最後に、次へ進めるか、追加調査が必要か、事業判断待ちかを明示する。

これにより、エージェントが勝手に実装へ進みすぎることを防ぎ、ユーザの事業判断境界を保てる。

## skill化するときの候補

将来 skill 化する場合、仮の skill 名は次のどれかがよい。

- `capability-readiness-assessment`
- `release-readiness-capability-mapping`
- `fit-gap-release-readiness`

### 起動条件

次のような依頼で起動できるとよい。

- リリース前に完成度を評価したい
- 既存コードベースの Fit & Gap を棚卸ししたい
- 業務ワークフロー単位で不足機能を整理したい
- ISMS、監査、セキュリティ、業務SaaSのMVP準備度を見たい
- 実装に入る前に P0/P1/P2/P3 へ分けたい

### skill が最初に読むもの

skill 化する場合、最初に読むものは以下を標準にする。

- `AGENTS.md`
- `README.md`
- `package.json`
- `docs/01-business/spec-dsl/`
- `docs/02-project/release-readiness/`
- `app/`
- `lib/`
- `tests/`
- `scripts/`

### skill の停止条件

skill は、評価作業と実装作業を分ける。

評価モードでは、実装に入らず、次のいずれかを記録して停止する。

- 次のGoalへ進める
- 追加調査が必要
- 事業判断待ち
- 実装前に品質ゲートや証跡定義を見直す必要がある

## 注意点

この方法では、次の誤判定を避ける。

- 画面があるだけで実務利用可能とみなす
- docsに書いてあるだけで実装済みとみなす
- APIやDBがあるだけで証跡が残るとみなす
- テストがあるだけで業務ジャーニー成立とみなす
- 総合点が高いだけでリリース候補とみなす
- エージェントが事業判断を実装判断として処理する

## 今後の検証観点

この方法が製品品質を上げる型として有効かは、今後の反復で検証する。

見るべき指標は以下。

- リリース前に P0/P1 を発見できた数
- CAPごとの証跡充足率
- Fit & Gap から実装バックログへの変換しやすさ
- 事業判断待ちが明確になった数
- 再実行時に同じような分類結果になるか
- 実装後の業務ジャーニー検証で失敗が減るか
- ユーザが次に何を判断すべきか分かりやすくなるか

## 現時点の位置づけ

この方法は、まだ正式な社内標準や完成済み skill ではない。

現時点では、ISMS Pilot の Goal 1〜6 を通じて検証している作業仮説である。Goal 6 完了後、実際に再利用できる粒度になっているかを見直し、skill 化するか、project runbook として残すか、両方に分けるかを判断する。
