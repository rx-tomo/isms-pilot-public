1. 全体像: 実装から参照仕様へ

仕様パッケージは、現行コードに埋め込まれた業務仕様を、AIや外部システムが検証しやすい単位に分解したものです。正本はコードとDB定義ですが、仕様レビューや連携設計ではこのパッケージを入口にします。

入力ソース DBスキーマ、サービス層、API Route、業務docs、QAシナリオ
→
仕様パッケージ process、decision_rules、data_schema、api_spec、exception_policy、glossary
→
利用者 エンジニア、QA、AIエージェント、外部連携、監査/仕様レビュー

ファイル構成と役割

2. 業務フローの中核

Riscala AI for ISMSは、テナント/組織設定を起点に、情報資産、リスク、文書、監査、マネジメントレビューへ流れるISMS運用サイクルを持ちます。

1組織設定
スコープ、部門、ユーザー、ロール、権限
2資産・リスク
情報資産、リスク評価、対応、管理策
3文書統制
作成、版管理、2段階承認、エクスポート
4内部監査
計画承認、チェック、不適合、報告書承認
5改善
フォローアップ、マネジメントレビュー、次サイクル

文書承認のスイムレーン概要

作成者
下書き文書を作成し、一次承認者と最終承認者を選択して承認依頼を開始する。
System
文書状態を draft から in_review へ変更し、approval_requests と通知、監査ログを作成する。
承認者
現在のpending承認者だけが承認/却下できる。却下時は後続pendingも却下され、文書は draft に戻る。
System
全承認が完了すると文書を approved にし、approved_by と approved_at を記録する。

3. 判断ルールと状態遷移

判断ルールはDMN風YAMLにまとめられています。実装確認済みの条件分岐を、入力、出力、HTTPステータス、エラー文言に分解しています。

文書状態

draft承認依頼可能
↓ submit
in_review承認中。却下でdraftへ戻る
↓ all approved
approved承認済み
↓ obsolete
obsolete廃止扱い

承認リクエスト状態

pending承認待ち。期限超過ならエスカレーション対象
↓ approve / reject
approved差戻しでpendingへ
rejected差戻しでpendingへ
expired状態値は定義済み

5. 例外処理・監査観点

例外ポリシーは、エラー時に何を返すか、ログに何を残すか、どこで人手確認へ戻すかを整理するための入口です。

自動処理できない/戻す条件

条件戻し先
文書承認者未指定承認依頼モーダルで再選択
文書却下作成者が文書を修正し再申請
監査計画却下監査計画を修正し再申請判断
AI提案採用/不採用を人が判断
インポートエラー行CSV/ZIP内容を修正して再投入

監査ログに含めるべき情報

  • organization_id
  • user_id または actor_id
  • action または event_type
  • resource_type と resource_id
  • changes または payload
  • created_at

確認済みイベント語彙

領域イベント
documentdocument.created, document.approval_requested, document.approved, document.rejected
approval_eventrequested, approved, rejected, escalated, reverted
auditaudit.plan.approval_requested, audit.plan.approved, audit.report.approved, audit.report.rejected
riskrisk.created, risk.updated, risk.deleted
permissionuser.permissions_updated

未確認として残すべきもの

通知再試行
外部チャネル配信失敗時の再試行回数、バックオフ、停止条件は未確認。
失敗系監査ログ
すべての失敗を監査ログへ残す方針かは未確認。
課金上限
要件上の上限/自動アップグレードが実装済みか追加照合が必要。
全APIの外部公開範囲
画面だけが使うサービス操作と外部API化する操作の境界は未確認。

6. エンジニア向け活用ガイド

改修前チェック

業務フロー変更

QA/AI検証への使い方

判断表からテスト生成
DMN風ルールの各行を正常系/異常系テストケースへ展開する。
スキーマ検証
主要DTOをJSON Schemaで検証し、必須項目や状態値の揺れを検知する。
未確認事項の棚卸し
仕様レビュー時に未確認リストを潰し、推測が実装へ混ざらないようにする。
