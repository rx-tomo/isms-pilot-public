# ビジネス要件の確認事項と対応状況

本ドキュメントは、**Riscala AI for ISMS** の事業・課金・運用に関する未決事項を管理します。各項目の状態と次アクションを明記し、非エンジニアのステークホルダーでも判断できる粒度で整理しています。数値や仕様が確定した場合は、関連ドキュメント（例: `docs/01-business/requirements.md`、`docs/01-business/spec-dsl/`、`docs/10-improvement-plan/`）にも反映してください。

## 0. 2025-12-01 決定事項サマリー

| 項目 | 決定内容 |
| --- | --- |
| サービス正式名称 | **Riscala AI for ISMS** |
| 価格戦略（年間契約・割引） | MVP期間は月額のみ提供。年間契約・割引は正式リリース後に導入 |
| SLA指標 | MVP期間はSLA非公開。運用実績6ヶ月後に策定 |
| API公開ポリシー | 顧客要望3社以上でOAuth対応の外部API公開を優先対応 |
| カスタマーサクセス体制 | セルフサービス中心（FAQ/動画/ヘルプセンター）、複雑な問い合わせのみメール対応 |
| 24/365サポート | 営業時間内サポートのみ（平日9:00-18:00 JST）、緊急対応は翌営業日 |
| インシデント対応体制 | 3段階エスカレーション（開発チーム初動→経営判断→外部専門家招聘） |
| AIエンジン選定 | 外部API（OpenAI/Claude等）を利用。データプライバシー配慮を設計に組み込む |
| モバイルアプリ | 不要。レスポンシブWebデザインで対応 |
| オンプレミス版 | Phase 3以降で顧客要望を見て検討。現時点では決定保留 |
| パートナー戦略 | MVP期間は直販のみ。正式リリース後にパートナープログラムを設計 |

## 1. 組織・ユーザー管理
| 項目 | 状態 | 現状 / 次のアクション |
| --- | --- | --- |
| 1.1 料金プラン上限超過時の扱い | ✅ 方針確定 | 上限到達時に管理者へ通知し、そのまま継続利用できるよう即時課金（座席追加またはプラン自動アップグレード）を適用する。70〜90%到達アラートと Stripe Billing でのメータード課金設計を実装する。 |
| 1.2 組織登録情報の収集範囲 | ✅ 確定 | `organizations` テーブルで業種（文字列）、従業員数区分（5 段階）、ISO 取得状況（4 区分）を保持済み。入力フォームと翻訳キーを整理する。 |
| 1.3 マルチテナント・部門管理 | ⚙ 方針確定（実装準備） | 単一組織内で部門階層を持たせて権限・表示を分割する。`department_path`（または `parent_department_id`）を追加し RBAC を強化。将来的にサブ組織化が必要になった場合は階層の親ノード昇格で対応する。 |

## 2. 文書管理
| 項目 | 状態 | 現状 / 次のアクション |
| --- | --- | --- |
| 2.1 文書テンプレートの初期提供数 | ⚙ 方針確定（拡充中） | 規格が要求する主要テーマに沿った 10 種類（情報セキュリティ方針、アクセス管理、BCP、インシデント対応、教育訓練、リスク評価、内部監査、サプライヤ管理、資産台帳、例外管理）を用意する。既存シードへ順次追加する。 |
| 2.2 承認フローの段階数 | ✅ 方針確定 | 2 段階承認（作成者 → 上長 → ISMS 責任者）を採用。上長と責任者が同一の場合はステップを自動スキップできるようワークフロー実装する。 |
| 2.3 バージョン履歴の保持期間 | ⚙ 方針確定（実装準備） | システムとしては無制限保存を許容。最新 5 件を超える履歴は追加課金対象とし、契約終了 90 日後に自動削除。即時削除 API も提供する。 |

## 3. リスクアセスメント
| 項目 | 状態 | 現状 / 次のアクション |
| --- | --- | --- |
| 3.1 リスク評価尺度（影響度 × 発生可能性） | ✅ 確定 | `create_default_risk_criteria` で 5 段階定義済み。レベル説明文を UI とドキュメントに表示する。 |

## 4. 監査機能
| 項目 | 状態 | 現状 / 次のアクション |
| --- | --- | --- |
| 4.1 ISO 27001 チェックリストの詳細度 | ⚙ 方針確定（シード整備中） | 附属書 A 114 項目を個別管理。`iso27001_requirements` に Clause/Control/Objective を保持し、必須・任意タグで運用する。 |

## 6. セキュリティ・コンプライアンス
| 項目 | 状態 | 現状 / 次のアクション |
| --- | --- | --- |
| 6.1 データ保存場所 | ⚙ 方針確定（今後拡張） | 現行は Drizzle + libSQL/Turso を標準とする。ローカルは `file:local.db`、クラウドは Turso を想定し、リージョンやバックアップ方針は商用準備時に確定する。 |

| 6.3 バックアップ頻度と RPO/RTO | ⚙ 方針確定（詳細設計中） | 日次バックアップ＋ PITR を前提に RPO 24 時間以内を確保。復旧手順と定期テスト計画を策定する。 |
