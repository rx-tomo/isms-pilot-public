# Approval Responsibility Matrix

この文書は、`approver` ロールを単一の承認者として扱いすぎないための整理である。現行実装では `approval_requests` / `approval_events` が汎用承認基盤になっているが、ISMS実務では承認対象ごとに責任者、根拠、必要証跡が異なる。

## ロール整理

| 種別 | 役割 | 備考 |
| --- | --- | --- |
| ログインロール | `approver` | 承認キューや文書レビューなど、システム上の承認操作を許可するためのロール。 |
| 体制ロール | CISO、情報セキュリティ責任者、ISMS管理責任者、リスクオーナー、部門責任者、経営層 | どの承認対象を誰が責任を持って判断するかを表す業務ロール。 |
| 監査ロール | `auditor` / 主任監査員 / 監査チームメンバー | 内部監査を実施・記録する側。報告書を作成するが、報告書の最終承認者とは限らない。 |

## 次に実装/QAする候補

1. 残留リスク受容の多段承認: CISO代表ルールと責任者本人ルールは通ったため、次はリスクオーナー/CISO/経営層を直列または条件分岐にする必要があるかを整理する。
2. 是正完了承認: `nonconformity_closure` として承認キュー接続済み。次はCISO単独承認で足りるか、監査責任者と情報セキュリティ責任者を分けるかを実務検証で判断する。
3. マネジメントレビュー判断承認: 経営判断の正式化が必要と判断された場合に、W-05で扱う。
4. 審査準備パッケージ: 監査計画、監査報告書、是正、リスク、管理策、文書を審査準備に使える証跡パッケージへまとめる。

## 実務検証で確認する問い

- 承認者はログインロール `approver` で十分か、体制ロールから自動解決すべきか。2026-06-08時点では、残留リスク受容の責任者本人を承認者にする代表ルールはQA済み。
- 承認対象ごとに、承認理由、コメント、期限、再レビュー日、差戻し理由をどこまで必須にするか。
- 審査で説明する証跡として、`approval_events` と `audit_logs` のどちらを正本にするか。
- `auditor` が作成した監査報告書を、誰が承認すればISMS上妥当か。
- 残留リスク受容は、責任者本人承認の後に、CISO承認、経営承認のどこまで必要か。
