# Parent Objective

## Fixed Parent Objective

Riscala AI for ISMSを、商用公開前の実務検証版として、未認証企業の初回審査登録準備と、認証済み企業の1年間の継続運用を、自分が利用者・テスターとして試せる状態に近づける。

このObjectiveは開発上の中間ゴールである。プロダクトの最終ゴールは、初回審査登録準備と認証後1年間の運用を支援するSaaSサービスとして完成させることにある。PR/FAQ、顧客向け文言、サービス内メニューでは、内部の実務検証プロセスをそのまま顧客価値として表現しない。

この親目標は、単に画面やAPIが存在する状態ではなく、ISMS新規導入、継続運用、内部監査、経営レビューの主要業務が、利用者の操作、DB/API、監査ログ、出力物の証跡で説明できる状態を目指す。

当面の成功条件は商用公開や課金運用の完成ではない。ユーザ自身がテスターとして使い、業務に役立つか、どこで詰まるか、何が過剰または不足かを判断できる状態にする。

## Success Boundary

- `initial` ストーリーで、未認証企業が初回審査登録に向けて、スコープ、体制、文書、資産、リスク、管理策、初期タスクを準備できるか検証できる。
- `surveillance` ストーリーで、認証済み企業が1年間の継続運用、内部監査、マネジメントレビュー、是正、継続改善を回せるか検証できる。
- W-02〜W-05の主要業務ジャーニーを、画面/API/DB/tests/ログ/出力物の証跡で検証できる。
- 詰まりを、仕様不足、実装不足、テスト契約ズレ、事業判断待ちに分類できる。
- `npm run qa:security` でcritical/high/OSV findingsが未処理で残らない。
- CAP-01〜CAP-32のP0/P1が、実装済み、検証済み、判断待ち、後続化のいずれかに分類されている。

## Non-Goals

- このGoalだけで商用公開、課金運用、顧客配布、利用規約整備を完了すること。
- 認証取得や審査合格を保証する表現を作ること。
- 事業判断なしにデータ保持、物理削除、契約終了時責任範囲を決めること。
- P0/P1が残る状態でP2/P3の見栄え改善や高度AI連携へ進むこと。
