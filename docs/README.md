# Documentation Workspace Guide

作成日: 2025-06-02 (tom)
更新日: 2025-12-01
記録者: Codex

このリポジトリでは、プロダクト開発に必要なナレッジを `docs/` に体系化して保存します。本ガイドは、他プロジェクトでも再利用できる汎用構成の概要と運用ルールをまとめたものです。プロジェクトを新規に立ち上げる際は、この構成をテンプレートとして調整してください。

## ディレクトリ構成

```
docs/
├── 01-business/        # 事業要件・市場分析・KPI
├── 02-project/         # 計画・進捗管理・ユースケース
├── 03-architecture/    # システム設計・技術決定
├── 04-development/     # コーディング規約・実装ガイド
├── 05-quality/         # テスト・QA プロセス
├── 06-operations/      # 環境構築・運用手順
├── 07-design-system/   # デザインシステム・UI指針
├── archive/            # 参照用の旧資料・実験メモ
├── handoff/            # 日次/週次ハンドオフ記録（*.yaml）
├── mockups/            # 画面モックやワイヤーフレーム
└── README.md           # 本ガイド
```

### フォルダ別の想定内容
- **01-business/**: 事業戦略、顧客ペルソナ、料金モデル、認証・規制要件など、プロダクトの背景情報を集約します。
- **02-project/**: ロードマップ、フェーズ計画、ウォーキングスケルトン方針、進捗レポートなど、プロジェクトマネジメントに関するドキュメントを管理します。
- **03-architecture/**: アーキテクチャ図、DB 設計、RBAC/セキュリティ方針、技術調査の結果を保存します。変更時は関連するマイグレーションや設計判断のログも併記します。
- **04-development/**: ディレクトリ構成ポリシー、命名規則、翻訳ガイドラインなど、開発者が日常的に参照する規約をまとめます。
- **05-quality/**: テストケース、QA チェックリスト、自動テストスクリプトの説明、リリース前チェックリストなど品質保証に関する資料を置きます。
  - `uc/UC-01-onboarding/qa-plan.md` 〜 `UC-10-tenant-provision/qa-plan.md` でユースケース単位のテスト観点と Playwright/CLI コマンドを管理します（テンプレート: `docs/05-quality/uc/README.md`）。
  - オペレーター向け手順書は `qa-ucXX-<role>-operator.md` として保存し、`qa-plan.md` からリンクします。
- **06-operations/**: ローカル/ステージング/本番環境のセットアップ手順、依存サービスの設定、運用上の注意点を記録します。
- **07-design-system/**: デザイン原則、トークン設計、UI コンポーネント仕様、アクセシビリティ指針などを整理します。
- **archive/**: 過去の検討資料や廃止ガイドを保管します。現行運用に使用しないものはここへ移し、参照理由を明記します。
- **handoff/**: 日次・週次のハンドオフを YAML 形式で保存し、`project_state` や `open_items` を定型化して共有します。
- **mockups/**: HTML/PDF/画像などの静的モックを配置します。最新版がどれか識別できるようファイル名に日付やバージョンを含めてください。

## 運用ルール
1. **変更トレーサビリティ**: ドキュメントに記載した判断はソースコードやチケットと相互参照できるようリンクを追加します。
2. **バージョニング**: 定期レポートやスナップショットは日付をファイル名に含め、最新情報への導線（リンクや README 内の一覧）を用意します。
3. **翻訳・多言語**: プロジェクトで多言語対応が必要な場合、言語別の差分は同じファイル内にセクション分けするか、`messages/` 等の翻訳資産と同期を取ります。
4. **ハンドオフの標準化**: `handoff/YYYYMMDDHHmm.yaml` のようにタイムスタンプを付与し、`project_state`・`current_changes`・`open_items`・`next_steps` などのキーを共通化します。
5. **テンプレート活用**: 新しい成果物を追加する際は、既存ファイルの構成をコピーし、プロジェクト固有の内容に置き換えます。必要に応じて `docs/templates/` フォルダを作成し、共通テンプレートを格納してください。

## QA/テストドキュメントのトレーサビリティ

1. **MVP 確認**: 実装がロール別 MVP を満たしたら `docs/02-project/12_uc-checklist.md` の該当項目を更新します。
2. **QA Plan 更新**: 各 UC の `docs/05-quality/uc/UC-01-onboarding/qa-plan.md` 〜 `UC-10-tenant-provision/qa-plan.md` に設計リンク、観点 (Given/When/Then)、CLI/Playwright コマンド、オペレーター手順書リンクを記載します。
3. **自動テスト**: `npm run qa:<uc>` や `npm run e2e:<uc>` の結果ログを `qa-plan.md` に反映し、観点ごとの自動化状況を管理します。
4. **オペレーター検証**: `docs/05-quality/qa-ucXX-<role>-operator.md` を更新し、実施ログは `docs/02-project/reviews/YYYYMMDD_<topic>.md` に追記します。
5. **改善フィードバック**: レビュー/改善メモ（`docs/02-project/reviews/` や `docs/02-project/90_*.md`）を `qa-plan.md` の「フィードバック履歴」から参照できるようリンクします。

このフローを厳守することで、「どのユースケースがどこまで検証済みか」「どのテスト資産に落とし込まれているか」を後からトレース可能にします。

## カスタマイズの指針
- プロジェクトの性質に合わせてサブフォルダを追加する場合は、数値プレフィックス（`08-security/` など）を付けるとソート順が安定します。
- 他チームと共有する場合は、`README.md` の「運用ルール」と「フォルダ別の想定内容」を各チームの責務に合わせて編集し、参照先リソース（Issue 管理ツールや外部 Wiki 等）へのリンクも併記してください。
- ドキュメントの更新頻度が高いテーマは、担当者や更新周期を記載した `MAINTAINERS.md` を用意すると維持管理が容易です。

## 導入時チェックリスト
- [ ] プロジェクトに合わせたフォルダ構成と命名規則を決定した
- [ ] 主要ドキュメント（要件・ロードマップ・設計・品質・運用）が初期化されている
- [ ] ハンドオフテンプレートを整備し、共有フローをチーム内で合意した
- [ ] 他リポジトリや外部ツールと参照関係を整理した
- [ ] ドキュメント更新時のレビュー手順を決めた

この README はプロジェクトの成長に合わせて随時更新してください。新しいカテゴリや運用ルールが決まった際は、本ガイドに追記し、他の関係者が迷わず参照できるようにしておきます。
