# ISMS Pilot 公開版の概要

ISMS Pilot は、ISO/IEC 27001 に基づく ISMS の構築、認証準備、認証後の継続運用を支援する Web アプリケーションです。

この文書は、GitHub 上で初めて読む人が「何のプロダクトなのか」「今どこまでできているのか」「なぜ公開しているのか」を日本語で把握できるようにまとめた公開向け概要です。

## 公開している目的

ISMS Pilot の公開は、商用 SaaS サービスの正式開始ではありません。開発途上のプロダクトを外から見える形にし、ISMS 実務者、SaaS 開発者、AI 駆動開発に関心のある人、協業や貢献の可能性を検討する人からフィードバックを得るための Build in Public です。

もうひとつの目的は、AI 駆動開発の実証です。プロダクトオーナーは要求、優先順位、事業判断、レビュー判断を行い、アプリケーションの実装、修正、検証、ドキュメント整備は AI エージェントとの協働で進めています。オーナー自身はアプリケーションコードを 1 行ずつ直接書いていません。

この公開は、ISMS 支援アプリケーションとしての方向性と、AI エージェントを使った開発スタイルの両方を検証するためのものです。

## ISMS を知らない人向けの説明

ISMS は、会社や組織が情報セキュリティを継続的に管理するための仕組みです。ISO/IEC 27001 は、その仕組みを作り、運用し、監査を受けるための国際規格です。

ISMS の運用では、情報資産の整理、リスク評価、管理策の選定、文書の作成、教育、内部監査、是正処置、経営層への報告など、多くの作業が継続的に発生します。ISMS Pilot は、それらを別々のファイルや表計算シートで管理するのではなく、ひとつのアプリケーション上でつなげて扱うことを目指しています。

## ISMS 実務者向けの説明

ISMS Pilot は、初回認証の準備だけでなく、認証後の年次運用を想定しています。適用範囲、組織、ユーザー、文書、情報資産、リスク、リスク対応、管理策、適用宣言、教育、内部監査、是正処置、マネジメントレビュー、証跡出力を、同じ業務フローの中で扱います。

特に、管理策の適用判断、判断理由、版管理、レビュー、承認、再発行を、単なる文書ではなく業務データとして扱えるようにすることを重視しています。

## 現在確認できる主な機能

### SaaS 型の利用者モデル

現在の実装は、商用 SaaS を想定した深いユーザーモデルを持っています。サービス運営者、テナント管理者、テナント利用者のような階層を前提に、組織ごとの利用、権限、プラン、課金連携を扱える方向で設計されています。

将来、単一利用者向けや個別導入向けに切り出す場合は、この深いユーザー管理モデルを簡略化して使う可能性もあります。

### ISMS 初期構築と認証準備

- 適用範囲、組織情報、利用者、拠点、業務対象の整理
- 情報資産の登録と分類
- リスク評価、リスク対応、残留リスクの整理
- 管理策の選定と適用判断
- 適用宣言に近い情報の管理
- 認証審査に向けた文書や証跡の準備

### 認証後の年次運用

- 年次の運用サイクルを意識したタスク管理
- 内部監査の計画、実施、指摘、是正処置
- 是正処置の期限、フォローアップ、完了確認
- マネジメントレビューの記録
- リスク再評価、残留リスクの見直し
- 次回審査に向けた証跡の整理

### 文書、証跡、エクスポート

- 文書の登録、版管理、承認状態の管理
- 監査やレビューで使う証跡パッケージの出力構想
- CSV、Excel、Word で扱いやすい形式へのエクスポート方針
- 組織データのバックアップ ZIP 出力方針

SaaS に登録した情報を閉じ込めず、必要なときに手元へ取り出し、通常の ISMS 運用に戻せることを重視しています。逆に、既存の Excel や Word ベースの運用資料を取り込む方向も想定しています。

### タスク、通知、協働

- ISMS 運用上のタスク登録
- 期限、担当、状態の管理
- コメント、添付、タグ、通知の設計
- 監査、教育、是正処置、レビューとタスクの連携

### 課金と外部サービス連携

現在の設計には、テナントごとのプラン、サブスクリプション、オンライン決済サービスとの接続を想定した要素が含まれています。ローカル評価ではモックモードで動かす前提です。

実際の商用提供に必要な価格、契約、請求運用、サポート、SLA はまだ最終決定ではありません。

### ローカル実行とデータベース

ローカルでは SQLite/libSQL 互換の `local.db` を使って動作します。クラウド DB としては Turso を想定しています。DB アクセスは Drizzle ORM と libSQL クライアントを使います。

## まだ完成していないこと

この公開版は、完成済みの商用サービスではありません。特に次の領域は、今後の検討または追加実装の対象です。

- 商用提供時の価格、契約、利用規約、SLA、サポート範囲
- 本番運用時の監視、バックアップ、障害対応、RTO/RPO
- 本番セキュリティレビュー、権限設計、監査ログ、個人情報保護
- 実顧客データの移行、既存 Excel/Word からのインポート精度
- 単一利用者向けパッケージとして切り出す場合の設計
- 外部コントリビューターを受け入れる場合のルール
- 商用利用、再配布、ホスティング利用、派生サービス利用の許諾条件

また、このアプリケーションを使うこと自体が ISO/IEC 27001 の認証取得や維持を保証するものではありません。

## 元リポジトリと公開リポジトリの関係

開発の正本は非公開の元リポジトリです。公開リポジトリは正式な fork ではなく、公開してよいファイルだけを抜き出した snapshot です。

- 元リポジトリ: `rx-tomo/pjt007`
- ローカルの元リポジトリ: `/Users/tom/workspace/work/pjt007`
- 公開リポジトリ: `rx-tomo/isms-pilot-public`

同期では、元リポジトリの Git 履歴を公開リポジトリへ push しません。必要なタイミングで、公開対象のファイルだけを allowlist 方式で公開用 checkout にコピーし、内部メモ、handoff、archive、meeting notes、ローカル DB、ログ、秘密情報候補、生成物を除外してから、公開リポジトリ側で検証、commit、push します。

そのため、元リポジトリには内部開発の完全な文脈が残り、公開リポジトリには外部の読者が読んでよいコードと文書だけが残ります。
