# ISMS必要能力ギャップ評価

## 結論

現行システムは、文書、リスク、タスク、監査、テナント運用の実装資産を広く持つ。一方で、リリース完成度を評価するには「存在する」だけでは不十分であり、ISMS実務工程に沿った一気通貫確認、テナント分離の否定ケース、セキュリティ脆弱性解消、unit/E2E復旧が必要である。

2026-05-14の後続Goal 1調査により、教育・力量、供給者/クラウド、KPI/監視測定、リスク基準/残留リスク受容、管理策運用証跡、外部審査証跡、変更管理、復旧/BCPを追加候補として扱うことにした。Goal 2では、CAP-01〜CAP-24だけでなくCAP-25〜CAP-32もFit & Gap対象に含める。

Goal 6では `qa:rbac:matrix` 34 passed と `npm run typecheck` passを維持し、`npm run test:unit`、`test:e2e:smoke`、`qa:documents`、主要APIのtenant isolation否定E2Eを復旧した。その後のNext 16 major upgradeにより、2026-05-15 `npm run qa:security` はcritical 0 / high 0 / OSV findings 0となり、CAP-23のP0 security high gateは技術的に解消済みである。

## 90%到達に必要な変更

1. `qa:security` のhighはNext 16 major upgradeで解消済み。release-readiness docs/scoreへ反映し、`no_high_or_critical_security_issue` はpassへ更新済み。
2. W-01〜W-06の代表ジャーニーで、RBAC/tenant isolationの否定E2Eに加えて監査ログと実務証跡を確認する。
3. `npm run test:unit` は復旧済みだが、runnerから一時除外した `webhook-delivery.test.ts` と `ai-suggestion-repository.test.ts` を後続で正規復帰する。
4. `test:e2e:smoke` は復旧済み。W-01〜W-06ではsmokeより深い業務フローを検証する。
5. `qa:documents` は復旧済み。文書・エクスポート・証跡探索は業務ジャーニー内で確認する。
6. CAP-01〜CAP-24の未検証項目を、コード確認だけでなく画面/API/QA証跡に結びつける。
7. データ保持/論理削除/契約終了時運用をPO決定とdocsで正本化し、判断待ちをリリース阻害か後続改善かに分ける。
8. P2/P3項目は、MVP必須範囲から外す場合も理由と残リスクを明記する。
9. CAP-25〜CAP-32をGoal 2で棚卸しし、MVP必須/重要/後続/判断待ちへ分類する。
10. CAP-09/CAP-10/CAP-13/CAP-15/CAP-16/CAP-18は、Goal 1補正後の証跡定義に沿って再評価する。
