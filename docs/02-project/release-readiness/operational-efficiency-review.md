# ISMS運用効率レビュー

## 位置づけ

工程5「効率性レビュー」は、ページやマニュアルの存在確認ではなく、利用者がISMS運用を効率よく進められるかを評価する工程である。

## 評価シナリオ

| シナリオ | 期待する効率性 | 現状判定 |
| --- | --- | --- |
| 初回導入 | 組織設定、スコープ、ユーザー招待、文書、資産、リスク、管理策、タスクの次アクションが迷わず分かる | 未検証 |
| 月次運用 | 未承認、期限超過、リスク見直し、タスク進捗、通知が一画面または少ない操作で把握できる | 未検証 |
| 内部監査準備 | 必要文書、リスク、証跡、チェックリスト、前回指摘を探しやすい | 未検証 |
| 不適合/是正 | 指摘から是正タスク、期限、フォローアップまで追跡できる | 未検証 |
| SaaS運営 | テナント作成、ロック、契約状態、サポート調査、監査ログ確認が安全に短時間でできる | 未検証 |
| 契約終了/データ保持 | エクスポート、保持期間、論理削除、アクセス制御の運用が明確 | 判断待ち |
| AI支援 | AI提案を人が確認し、採否と根拠が監査ログに残る | 未検証 |
| 教育・力量管理 | 役割別教育、受講記録、力量確認、内部監査員力量を少ない操作で確認できる | 未検証 |
| 供給者/クラウド管理 | 委託先評価、契約要求、定期レビュー、関連リスクを探しやすい | 未検証 |
| KPI/監視測定 | 情報セキュリティ目的、KPI、測定結果、改善アクションが経営レビューへ接続される | 未検証 |
| 審査証跡準備 | 初回審査、サーベイランス、再認証で必要な資料をまとめて出力・説明できる | 未検証 |

## 初回評価の不足

- `docs/08-user-manual/` と `docs/06-operations/` の存在は、効率性の証拠としては弱い。
- `build` 成功は、画面が利用者にとって効率的に使える証拠ではない。
- `qa:documents` とlocalhost確認がtimeoutしており、実画面の応答性・導線・迷いポイントを確認できていない。
- coming-soon、スクリーンショット不足、古いQA記録が残っている場合、利用者の自己解決性に影響する。

## 再評価の観点

| 観点 | 確認内容 |
| --- | --- |
| 次アクション性 | ログイン後、対象ロールが次にすべき作業を理解できる |
| 一気通貫性 | 初期導入から監査準備まで、主要作業が途切れない |
| 証跡探索性 | 監査に必要な文書、ログ、出力、履歴を短時間で探せる |
| 例外復帰性 | 入力不備、権限不足、外部連携失敗、期限超過から復帰できる |
| 運用負荷低減 | テンプレート、CSV、通知、リマインド、自動タスク化が手作業を減らす |
| SaaS運営効率 | 運営者がテナントと契約状態を安全に管理できる |

## 補正判定

工程5の初回 `6/10` は過大評価である。現時点では、効率化機能の候補は豊富だが、実務シナリオに沿った操作確認ができていないため `3/10` を暫定値とする。
