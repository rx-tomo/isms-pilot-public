# 02-project 運用ガイド（フラット命名＋アーカイブ）

このディレクトリは「**アクティブな資料のみ直下に置く**」ことを徹底し、ファイル名の先頭番号で「抽象度（上流→下流）」と「着手順序」を表現します。完了したレポートやクローズ済みの検証ログは `archive/` サブフォルダへ移動し、同じファイル名で凍結保存します。

- 01–09: 計画（高抽象）
- 10–19: 運用（スプリント/実行）
- 90–99: ログ（履歴・スナップショット：**進行中のみ直下、それ以外は archive/**）

## 現在の主なファイル
- `01_roadmap.md` … 長期の道筋（いつ何を届けるか）
- `02_implementation-plan.md` … 実装計画（領域別の到達点・優先度）
- `03_user-journey.md` … 顧客体験の流れ（上流の基準）
- `10_plan-tracking.md` … 運用ハブ（進捗・次の優先タスク・リスク）。冒頭に **ロードマップ×機能ストリーム×UI/BE フェーズ** をまとめた俯瞰図と Mermaid 付きコンパクト概要を追加済み。
- `11_walking-skeleton-plan.md` … バーティカルな最小スライスの定義
- `12_uc-checklist.md` … ユースケース別チェックリスト（動作確認）
- `release-readiness/` … Agentic Engineering 前提のリリース完成度チェックシート、採点モデル、`/goal` 実行プロンプト
- `archive/README.md` … アーカイブ済みログの索引（目的・期間・補足を記載）
- `archive/90_2025-06-03_progress.md` ほか … 過去の進捗/検証ログを日付別に保存。2025年の `90_*` / `91_*` ログは原則ここへ集約

## 開発の考え方
このプロジェクトは新規事業×新規アプリケーションの 0→1 開発を前提にしています。既存システムのバグ修正や小規模改善ではなく、実際に UI/UX を触りながら価値検証を進めることを重視します。そのため、**デュアルトラック（Discovery と Delivery を並走）＋ウォーキングスケルトン**で折り合いをつけるワークフローを採用します。

### 押さえる原則（セオリー）
1. **デュアルトラックアジャイル**
   - *Discovery（学び）*: 事業仮説 → JTBD/主要ユースケース → プロト → ユーザ検証。
   - *Delivery（実装）*: 検証で確度が上がった要件から実装・運用へ。
   - → 「作ってから考える」を避け、**作る前に学ぶ**を並走させる。
2. **ウォーキングスケルトン（最薄の背骨）**
   - 認証/テナント/権限、イベントログ、課金など価値の通り道を**試験通電**させる。
   - Stripe/Supabase 課金はテストモード＋ダミープラン。ログはイベントスキーマだけ先に固定。
   - UI は仮でもよいので、**価値の最薄スライス**をエンドツーエンドで成立させる。
3. **アウトカム先行（KPI 定義）**
   - 例: 初回価値到達までの時間（TTFV）、1 週間継続率、有料転換率、NPS/タスク成功率。
   - 実装優先度は **アウトカムへの寄与** で決める。

### あなたの状況に合う現実的な折衷案
外部開発ベースでバックエンド先行志向を尊重しつつ、**UX 検証を止めない設計**にします。Discovery チームと Delivery チームを明確に分けるのではなく、ウォーキングスケルトンで合流点を作りながら前に進みます。

## 開発環境と本番環境の関係
- Twelve-Factor でいう **Environment Parity** を重視。ローカルと本番の差異は極小化。
- DB は Dump & Restore（Promotion）を前提にし、環境間の手動変更を禁止。

### 実務の型例
1. **スキーマは移行ファイルで管理**（マイグレーションが唯一の真実）
   - Supabase CLI / Prisma / Flyway / Liquibase 等でバージョン管理。
   - ローカルと本番の差分は手作業禁止、必ずマイグレーションで適用。
2. **データは役割で分けて移す**
   - スキーマのみ: `pg_dump --schema-only`
   - 基準データ（マスタ/設定）のみ: `pg_dump --data-only --table=public.master_* ...`
   - テスト・監査・イベント系は除外: `--exclude-table=public.event_* --exclude-table=public.audit_*`
3. **昇格（Promotion）手順を固定化**
   - (a) スキーマ適用 → (b) 基準データ投入 → (c) シード/初期ユーザ → (d) RLS/権限検証 → (e) スモークテスト。
   - Makefile / npm scripts / GitHub Actions などでワンコマンド化。

2. ロールごとに主要ユースケースを洗い出す（詳細化は後続で行う前提）。

## 関係性（上流→下流）
```
03_user-journey.md
  └─ 02_implementation-plan.md
       └─ 01_roadmap.md
            └─ 10_plan-tracking.md
                 ├─ 11_walking-skeleton-plan.md
                 │    └─ 12_uc-checklist.md

```
- 上流（03→02→01）の決定が下流に波及します。
- 10_plan-tracking.md は運用ハブ。方針変更は最初にここへ反映します。
- スケルトン/チェックリストはユースケース単位の最小スライスを同期管理します。
- 履歴は 90_* で凍結保存し、追記ではなく新規ファイルを追加します。

## アーカイブ運用と更新フロー

### 1. アーカイブルール
1. 進捗レポートや検証ログ（`90_*`）は「着手中 or 直近スプリントで参照するもの」のみ直下に置き、完了後は `archive/` に同名ファイルで移動する。
2. 計画ドキュメント（`01–12_*`）で大規模な改訂が完了し、以後参照しない旧版が発生した場合も `archive/` へ退避できる。最新版を必ず直下に維持する。
3. `archive/` への移動時は `10_plan-tracking.md` からの参照リンクを更新し、必要なら進捗サマリーに「アーカイブ済み」を明記する。
4. **タスク集約ルール**: ロードマップ・レビュー・設計ドキュメントで新たな未完タスクやチェックボックスが発生した場合、必ず `10_plan-tracking.md` の未対応事項（またはフェーズ別タスク表）へ転記し、元ドキュメントには「Plan Tracking #項目 を参照」と記載する（集約先は常に Plan Tracking とする）。Plan Tracking ではタスクを `B`（Business/PO 判断）、`I`（Implementation）、`D`（Design/UX）、`Q`（QA）などカテゴリタグで管理し、責務が一目で分かるようにする。

### 2. 更新フロー
1) 方針変更・新ユースケース追加
   - `03_user-journey.md` → `02_implementation-plan.md` → `01_roadmap.md` の順に更新。
   - `10_plan-tracking.md` のフェーズ/優先タスクを再定義。
2) スプリント開始
   - `10_plan-tracking.md` に目標とチェックポイントを記載。
   - `11_walking-skeleton-plan.md` を見直し、`12_uc-checklist.md` を同期。
3) 進捗共有
   - `90_YYYY-MM-DD_progress.md` を追加し、特記は `10_plan-tracking.md` から参照。
4) 検証/QA
   - 検証結果は 90_* に新規追加し、`12_uc-checklist.md` を更新。
5) 振り返り
   - 成果/課題/次アクションを 90_* でスナップショット化し、参照期間を過ぎたら `archive/` へ移動。必要に応じて上流（01–03）へ反映。

## このプロジェクトで重視する運用方針
- **フェーズ基準の同期**: 03 → 02 → 01 を順に更新し、`10_plan-tracking.md` と整合させてから実装へ進む。フェーズや優先タスクの変更は必ずロードマップとトラッキングを同時に更新する。
- **ウォーキングスケルトン最優先**: 新機能は最小スライスで縦に通してから横展開する。`11_walking-skeleton-plan.md` で UC ID ごとのスライスを言語化し、`12_uc-checklist.md` で動作確認を記録する。
- **履歴の透明性**: 実績や検証結果は `90_YYYY-MM-DD_*.md` を新規追加して凍結保存する。過去ファイルは編集せず、差分や補足は最新ログに追記する。
- **定期レビューとハンドオフ**: 週次レビュー/ハンドオフで `10_plan-tracking.md` を更新し、未決事項は `docs/01-business/business-requirements-open-questions.md` とリンクさせる。ハンドオフログは `docs/handoff/` に時系列で残す。
- **ユースケース ID による会話**: 計画・開発・検証の各場面で UC ID を使い、仕様や検証観点を迷わず辿れるようにする。

命名・関係性に変更が生じた場合は、本 README を最初に更新してください。
