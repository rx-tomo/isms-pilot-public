# i18n パリティチェック手順

トップページで `common.login.title` のようなキー文字列が露出した事象に対する、再発防止用の手順です。新規キー追加時やリリース前に本手順で検証・修復してください。

## 目的
- `messages/en.json` と `messages/ja.json` のキーセットを完全一致させ、ブラウザに生キーが表示される事故を防ぐ。
- 差分が出た場合の修復フローを標準化する。

## 実行タイミング
- 翻訳キーを追加・変更した PR 提出前
- リリース前の最終 QA
- i18n 関連のバグ修正後

## 手順（検知）
1. ルートで以下を実行する:
   ```bash
   node scripts/check-i18n-parity.js
   ```
2. `✅ en/ja message files contain the same key set...` が出ればパス。`Known missing ...` が残る場合は既知の負債リストなので無視可（ただし可能なら解消を検討）。
3. `❌ Keys missing ...` が出た場合は次の修復に進む。

## 手順（修復）
1. エラーに表示されたキーを `messages/en.json` と `messages/ja.json` の両方に追加する。
   - 既存の名前空間（例: `common.login`）に揃える。
   - 値は仮でも可だが、リリース前に自然な文言へ更新すること。
2. 追加したキーが大きな配列・オブジェクトに入る場合は、同セクション内でアルファベット順・論理グループ順を維持する。
3. 必要に応じて `scripts/check-i18n-parity.js` の `KNOWN_MISSING_*` から解消済みキーを削除する。
4. 再度 `node scripts/check-i18n-parity.js` を実行し、`✅` になることを確認。

## レビュー時のチェックポイント
- 新規コンポーネントで `t('...')` を呼んでいる箇所に対応する鍵が両言語で存在するか。
- UI でプレースホルダー（生キー表示）が出ていないかを軽く画面確認。
- `KNOWN_MISSING_*` に不要な借金が残っていないか。

## よくあるミスと対応
- **キーを片側だけ追加した**: パリティスクリプトで検出される。両方に追加する。
- **サブキー構造がずれた**: 例として `common.login` を文字列からオブジェクトに変更した場合、両言語で同じ構造に揃える。
- **負債リスト放置**: 修正できたら `KNOWN_MISSING_*` から削除し、スクリプト出力をクリーンに保つ。

## 保存先
- 本ドキュメントは `docs/05-quality/i18n-parity.md` に配置。変更時は PR の説明に追記し、関係者へ周知する。
