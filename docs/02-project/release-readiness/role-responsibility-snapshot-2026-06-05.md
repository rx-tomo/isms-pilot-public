# Role Responsibility Snapshot

記録日時: 2026-06-05 13:56:27 JST
追記日時: 2026-06-05 14:43:27 JST

このメモは、現時点の実装ベースで、操作者ロールの役割分担と代表的な操作を一旦固定するためのスナップショットである。今後の実務検証で、ロール名、メニュー名、承認対象、体制ロールとの対応は見直す。

この整理は「現時点の作業台帳」であり、最終仕様ではない。抽象的な部分や実務上の呼び名が合っていない部分は、後続の初回登録準備/継続運用QAで見つけた詰まりと一緒に見直す。

## 前提

- ここでの整理は、現行コードベースと会話内で確認した理解に基づく暫定版である。
- `auditor` は、ISMS文脈では内部監査に関するメンバーとして扱う。
- `approver` は、文書、内部監査、リスク受容、是正、経営判断など、複数種類の承認対象にまたがる可能性がある。
- 承認者の実体は、ログインロールだけでなく、情報セキュリティ責任者、ISMS管理責任者、CISO、部門責任者、経営層などの体制ロールと結びつけて再整理する必要がある。

## 現時点のロール整理

| ロール | 現時点の立場 | 代表的にできること | 主な未整理点 |
| --- | --- | --- | --- |
| `super_admin` / スーパー管理者 | SaaS全体の管理者 | テナント一覧、作成、ロック/解除、削除/復元、全体監査ログ、上位アカウント管理 | 商用契約、データ保持、復旧責任、課金運用との責任境界 |
| `system_operator` / システム運営者 | テナント運用を広く支援できるシステム側オペレーター | フェーズ選択、組織設定、初期セットアップ、ユーザー/権限、資産、リスク、タスク、監査、通知、課金導線の一部 | `super_admin` と `org_admin` の間の責任境界、顧客への提供運用時の権限範囲 |
| `org_admin` / 組織管理者 | 顧客企業内のISMS事務局・運用責任者 | 組織設定、ユーザー招待、体制、文書、資産、リスク、管理策、タスク、内部監査準備、課金導線の一部 | 初回登録準備から審査提出束までの一気通貫導線、年間運用の次アクション表示 |
| `user` / メンバー | ISMS運用に参加する現場メンバー | ホームで自分の作業確認、タスク進捗更新、文書確認、通知確認、部門スコープ内の運用参加 | 必読文書、教育、部門リスク/資産確認、インシデント/改善提案、部門スコープQA |
| `auditor` / 監査員 | 内部監査を実施・記録するメンバー | 監査ワークスペース、監査計画、チェックリスト、証跡確認、不適合、是正、フォローアップ、監査報告書 | 監査員力量、独立性、監査証拠の十分性、正式な報告書承認、審査提出向け証跡束 |
| `approver` / 承認者 | 重要な成果物や判断を承認するメンバー | 文書承認、承認キュー確認、レビュー対象タスク、リスク/是正/監査関連の承認候補 | 承認対象ごとの責任者定義、残留リスク受容、監査報告書承認、是正完了承認、経営判断承認 |

## 承認ロールの今後の整理方針

`approver` は単一のログインロールとして存在するが、実務上は承認対象ごとに意味が変わる。今後は次の粒度で整理する。

| 承認対象 | 想定される実務上の承認者 | 現状 |
| --- | --- | --- |
| 文書承認 | 文書オーナー、情報セキュリティ責任者、ISMS管理責任者、最終承認者 | 比較的実装が進んでいる |
| 監査計画/監査報告書 | 情報セキュリティ責任者、ISMS管理責任者、CISO | 監査計画と監査報告書は2026-06-05に承認申請、CISO承認、却下まで代表QA済み |
| 不適合/是正完了 | 是正責任者、情報セキュリティ責任者、監査責任者 | 2026-06-08追記: CISO代表での是正完了承認申請/却下/再申請/承認は代表QA済み。監査責任者と情報セキュリティ責任者を分けるかは後続判断 |
| 残留リスク受容 | リスクオーナー、情報セキュリティ責任者、CISO、経営層 | 承認者、理由、期限、履歴の構造化が不足 |
| マネジメントレビュー/経営判断 | 経営層、CISO、ISMS管理責任者 | 自由記述記録は可能、判断分類と承認履歴は不足 |

## 後で見直す対象

- ロール名: `system_operator`, `org_admin`, `approver` が実務者に伝わりやすいか。
- メニュー名: 監査、承認、タスク、マネジメントレビュー、リスク受容の入口が業務単位で分かりやすいか。
- 体制ロール: CISO、情報セキュリティ責任者、ISMS管理責任者、リスクオーナー、部門責任者をログインロールとどう結びつけるか。
- 承認種別: 文書承認、監査承認、是正承認、リスク受容承認、経営判断承認を同じ仕組みに載せるか、対象別に分けるか。
- 実務検証: 初回登録準備と継続運用の両ストーリーで、各ロールが迷わず次の操作に進めるか。

## 見直し時の扱い

- このスナップショットは、後からロール名、メニュー名、承認対象名を直すための比較基準にする。
- 変更するときは、単に名前を変えるだけでなく、画面表示、権限、API、監査ログ、spec-dslの用語を一緒に確認する。
- `approver` はログイン権限としては残しつつ、実務上の責任者は承認対象別に体制ロールへ寄せて整理する。

## 次に確認する観点

1. 6ロールをそのまま維持するか、表示名だけを実務寄りに変えるか。
2. `approver` を汎用承認者として残しつつ、承認対象ごとに体制ロールで制御するか。
3. 内部監査では、`auditor` と情報セキュリティ責任者/CISOの責任境界を画面とデータで明確にするか。
4. 初回登録準備ストーリーで、誰が何を承認すれば審査準備完了と言えるか。
5. 継続運用ストーリーで、年間イベントごとの責任者、確認者、承認者をどこまで構造化するか。
