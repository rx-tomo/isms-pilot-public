# ISMS実用化の整理・後続化候補

## 位置づけ

この文書は、過剰機能、未成熟機能、後続化候補、事業判断待ちを整理する。Goal 3時点では削除や非表示の実装は行わない。Goal 6で安全整理へ進む場合の正本候補とする。

## 結論

現時点で即削除すべき機能は確定しない。ISMS実務上の必要性があるものが多いため、削除よりも「既存CAPへ統合」「後続化」「判断待ち」「最小MVP化」で扱う。

特にCAP-25〜CAP-32は、独立機能として増やすと過剰になりやすい。CAP-28とCAP-29だけはP1として採用し、他はP2または判断待ちへ分ける。

## 整理候補

| 候補 | 分類 | 理由 | 推奨扱い |
| --- | --- | --- | --- |
| CAP-25の独立LMS化 | 後続化 | ISMS MVPには教育証跡で足りる。教材配信、テスト、修了証の高度化は重い。 | 削除ではなく後続候補。内部監査員力量だけCAP-13へ統合。 |
| CAP-27の専用KPI管理モデル | 統合 | 目的/KPIはレビューとダッシュボードに置いた方が実務導線が自然。 | CAP-15/CAP-20へ統合。 |
| CAP-29の汎用ファイル保管庫化 | 過剰候補 | 管理策・リスク・監査と紐づかないVaultは審査説明力が弱い。 | 関連CAP接続を必須条件にする。 |
| CAP-30の保証付き審査パッケージ表現 | 判断待ち | 認証取得保証や審査合格保証に見える表現は営業/法務判断が必要。 | 実装計画から分離し、証跡束エクスポートに留める。 |
| CAP-31をインシデント内に混在させる設計 | 整理候補 | 変更管理とインシデント対応は入力、承認、完了条件が違う。 | CAP-16から分離してP2 backlog化。 |
| CAP-32のSaaS復旧保証機能 | 判断待ち | バックアップ削除、RTO/RPO、責任境界に関わる。 | PO判断まで後続化。顧客BCP記録とは分ける。 |
| 高度AI自動判断 | 過剰候補 | AIは補助であり、最終判断は人間が行う前提。自動確定は責任境界が重い。 | 人手確認、採否ログ、privacy filterを最低条件にし、高度化はP3。 |
| Slack/Teams/高度Webhook連携 | 後続化 | 外部連携は運用効率に効くが、P0/P1ゲート解除より優先しない。 | P3。既存機能が壊れている場合だけP1化。 |
| 多言語/スクリーンショット補強 | 後続化 | リリース品質には有用だが、必須ゲートではない。 | P3。P0/P1解除後に扱う。 |

## 削除禁止・慎重扱い

| 対象 | 理由 |
| --- | --- |
| 認証、認可、RBAC、テナント分離 | SaaSとしての必須ゲート。根拠なく削除/緩和しない。 |
| 監査ログ、証跡、エクスポート | ISMS実務と審査説明力の中核。未成熟でも削除ではなく改善対象。 |
| 文書、リスク、管理策、内部監査、CAPA | ISMS主要導線。P0/P1解除後に一気通貫証跡化する。 |
| CAP-28/CAP-29 | Goal 1追加CAPだがP1へ昇格。後続化しない。 |

## Goal 6への引き継ぎ

Goal 6で過剰機能を整理する場合は、まずこの文書を読み、削除ではなく次の順で判断する。

1. 既存CAPへ統合できるか。
2. MVPの最小証跡だけ残せるか。
3. feature flagまたは非表示で安全に後続化できるか。
4. 事業判断待ちとして分離すべきか。
5. 本当に削除してよいか。
